# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Run

```bash
npm install          # first time only
npm run compile      # one-shot TypeScript compile → out/
npm run watch        # incremental watch (run before F5)
npm test             # compile + Node --test suites (out/**/ *.test.js)
```

`npm test` compiles, then runs every `*.test.js` under `out/{compare,core,doublepane,edit,preview}`. The edit suites that touch VS Code-only APIs (parser, navigator) load `edit/vscodeStub.js`, which shims `require('vscode')`, so the whole suite runs without an Extension Host. The `doublepane` suite is VS Code-free (like `actions.test.ts`), so it needs no stub.

CI runs `npm test` on every push and pull request ([.github/workflows/ci.yml](.github/workflows/ci.yml)); the release workflow ([.github/workflows/build.yml](.github/workflows/build.yml)) also runs `npm test` before packaging so no release is built from a red tree.

Press **F5** in VSCode to launch the Extension Development Host. Use **Developer: Reload Window** in the host to pick up changes without restarting the debugger.

Package for local install:
```bash
npx @vscode/vsce package
code --install-extension kaicrit-*.vsix
```

## Releasing

1. Bump `"version"` in [package.json](package.json) to the new version (e.g. `0.2.0`)
2. Update [CHANGELOG.md](CHANGELOG.md)
3. Commit: `chore(changelog): Prepare release vX.Y.Z`
4. Push a matching tag: `git tag vX.Y.Z && git push origin vX.Y.Z`

The CI workflow builds the `.vsix` with `@vscode/vsce package` passing the tag (minus its `v` prefix) as the **positional** version argument, so the published package always reflects the git tag. The `package.json` version is the canonical source for local development (shown when pressing F5).

## Architecture

The extension bundles four CriticMarkup features — **edit**, **compare**, **double-pane**, and **preview** — around one shared marker vocabulary. Source is grouped by feature under [src/](src/):

### `src/core/` — shared vocabulary

| File | Role |
|---|---|
| [core/types.ts](src/core/types.ts) | `ChangeType` const enum + `CriticChange` interface |
| [core/markers.ts](src/core/markers.ts) | `MARKERS` delimiter table + `RE_ALL` regex — the single source of truth for marker syntax, consumed by the edit parser and the compare renderer. The `findMarkers(text)` helper iterates markers via `text.matchAll(RE_ALL)`, which runs against an internal clone of the regex — so the shared global `RE_ALL` is never mutated and there is no `lastIndex` state shared between callers. Callers iterate with `for…of findMarkers(text)` instead of `RE_ALL.exec` |
| [core/comment.ts](src/core/comment.ts) | `parseCommentMeta(content)` — pure splitter for the optional `@author YYYY-MM-DD:` comment prefix, shared by the edit parser and the preview tokenizer (`core/comment.test.ts` covers it) |

| [edit/trackChangesEngine.ts](src/edit/trackChangesEngine.ts) | `computeTrackChanges(preText, rawEdits)` — pure, VS Code-free core of the Track Changes mode: classifies each raw edit (insert→`{++…++}`, delete→`{--…--}`, replace→`{~~…~>…~~}`) against surrounding markers and returns compensating replacements (post-raw-edit coords) + final caret offsets. Builds marker strings from `MARKERS`; single-edit adjacency-merges deletions, and (issue #34) **absorbs any edit that lands inside an existing marker's content** — typing/deleting inside an addition or a substitution's new side grows that side, editing a highlight/comment/deletion interior stays literal — so no nested markup can form. When the absorbed `newText` itself contains complete markers (pasting `{++any++}` into an addition's content), they are **flattened to their accept-form** by the shared `flattenInnerMarkers(text)` helper (`findMarkers` → `resolveReplacement(matchToResolvable(m), 'accept')` per inner marker, returns the flattened string or `null` when there is nothing to flatten) before being absorbed — the enclosing marker grows by the resulting plain text (`{++ananyy++}`) instead of nesting (`{++an{++any++}y++}`); plain typing with no marker stays a `skip`. The boundary counterpart (issue #38): a delete/replace (`oldLength > 0`) that **removes part of an existing marker's opener/closer**, while staying within that marker's bounds, is treated as **rejecting** the whole marker — it reuses the shared `resolveReplacement(…, 'reject')` from `edit/resolve.ts` (a `toResolvable` helper splits a substitution's `O~>N` content back into its two sides) and replaces the marker's post-raw-edit span with the reject result, instead of wrapping the touched delimiter as a new (nested) change. A pure insertion removes no delimiter, so it never rejects; an edit spilling past the marker bounds falls through to normal wrapping. The third nesting-prevention rule (issue #40): when the inserted `newText` **already contains complete CriticMarkup** (it parses to one or more markers via `findMarkers`), the markers are kept verbatim and only the surrounding plain runs are wrapped as additions — pasting `{++a++}` stays `{++a++}` (emitted as a `skip` when the text is *nothing but* markers), `foo {++a++} bar` becomes `{++foo ++}{++a++}{++ bar++}`, and a paste *over a selection* (`oldLength > 0`) tracks the replaced text as a leading `{--…--}` deletion. This check sits **after** the #34 absorption and #38 reject branches (so pasting inside a marker is handled by #34's flatten-on-absorb, and overwriting a delimiter still rejects) and **before** the plain insert/delete/replace wrapping; partial/unterminated input like `{++a` matches no marker and falls through to the normal addition wrap. The module also exports `computeNormalModeFlatten(preText, rawEdits)` — the **Track Changes OFF** counterpart that reuses *only* the #34 enclosing-content flatten (via the shared `flattenInnerMarkers`): it emits a compensating edit solely when a paste lands inside an existing marker's content and contains complete markers, and returns no edits for everything else (plain text stays plain, standalone pasted markup stays literal — #38/#40 and the addition/deletion/substitution wrapping are all skipped). Tested by `trackChangesEngine.test.ts` |
| [edit/trackChanges.ts](src/edit/trackChanges.ts) | `TrackChangesManager` — thin VS Code wrapper around the engine. Per-document state (`enabled` set + `shadow` text snapshots, since the change event omits deleted text), a **per-document** re-entrancy guard (`applyingOwnEdit` is a `Set<string>` of doc keys, not a single flag) so its own compensating `WorkspaceEdit` isn't re-processed — and so an in-flight `applyEdit` for document A never drops a concurrent edit in document B. The guard entry is added before the `applyEdit` and removed in **both** the success and the rejection branch of the returned promise, so a single failed `applyEdit` (read-only doc, conflicting edit) can't wedge the recorder permanently; the shadow-sync + caret-restore follow-up runs only when `applyEdit` resolves truthy. Undo/redo left untouched, plus the always-visible two-way `$(edit) Track Changes: On/Off` status-bar item (shown for `file`/`untitled` editors, click switches recording on **or** off — mirrors the `$(eye)` enablement toggle; refreshed by `syncActiveEditor` on activation and active-editor switch) and `kaicrit.trackChanges` context key. Before running the engine, `handleChange` plausibility-checks the `shadow` against the event geometry (every change's `rangeOffset + rangeLength` must fall within `shadow.length`); if an external mutation ever left the shadow out of sync it resyncs from the live document and skips wrapping that event, so a malformed marker can never be assembled from an inconsistent snapshot (issue #68). Takes an `isDocEnabled(doc)` predicate (from `EnablementManager`, like `DecoratorManager`) so the recorder is gated against the same language-whitelist / per-file decision as every other reader (issues #53, #54). `applyDefault` honours the `kaicrit.edit.trackChanges` setting for freshly opened docs **only when the document is kaicrit-enabled** — for a disabled doc it returns *without* marking the doc seen, so the default still applies the first time the doc later becomes enabled (issue #54). The `kaicrit.toggleTrackChanges` command (in `commands.ts`) implicitly enables kaicrit for the file (`em.toggle`) when turning recording ON in a disabled doc, so the recorded markup is never invisible/unresolvable text noise (issue #54). `applyResolution(doc, edit)` is the entry point `commands.ts`'s `applyAt`/`applyAll` use to apply an accept/reject `WorkspaceEdit`: it wraps `applyEdit` in the **same per-document `applyingOwnEdit` guard** so the recorder doesn't re-process its own resolution — without it, removing a marker's delimiters would re-trigger the engine's #38 delimiter-reject rule and silently undo the accept/reject (issue #42) — then refreshes the shadow snapshot to the post-resolution text (only while recording is on). `applyAuthoringEdit(doc, apply)` is the sibling entry point the `commands.ts` insert/wrap commands use: it wraps the supplied `apply()` (an `editor.edit` thunk) in the **same per-document guard** so explicit markup authoring lands verbatim and the recorder doesn't re-track the wrap as a replace — without it, wrapping a selection in a highlight/comment prepends a spurious deletion of the selected text (issue #44) — then refreshes the shadow when recording is on. When a document is **not** tracking, `handleChange` routes to `handleNormalMode`, which first takes a cheap early-out — it returns immediately unless some `contentChange.text` contains a `{` (the only way the flatten can ever fire), so ordinary typing skips the config read, the full-document `getText`, the pre-text reconstruction and the marker scan entirely (issue #53) — then skips disabled documents via `isDocEnabled` (issues #53, #54), and otherwise (gated on the `kaicrit.edit.preventNestingOnPaste` setting, default `true`) runs `computeNormalModeFlatten` to prevent nested markup on paste even with tracking off; it reconstructs the needed `preText` from the post-edit document plus the change event's `rangeOffset`/`rangeLength`/`text` (a same-length, delimiter-free filler stands in for the unknown deleted span — the engine never reads the deleted bytes), and reuses the **same** `applyingOwnEdit` per-document guard + applyEdit/caret-restore structure as the tracked path. Covered by `trackChanges.test.ts` (guard release on rejection + per-document isolation, the #42 `applyResolution` suppression + guard-release cases, the #44 `applyAuthoringEdit` suppression + guard-release cases, plus normal-mode flatten/passthrough/guard/undo cases, via the `vscodeStub`'s swappable `applyEdit` and `setConfig`) |

### `src/compare/` — diff two files into CriticMarkup

| File | Role |
|---|---|
| [compare/diff.ts](src/compare/diff.ts) | Myers diff over tokenized text → `DiffOp[]`; whitespace-preserving tokenizer (the `word` granularity uses Unicode property classes `[\p{L}\p{N}_]+|\s+|[^\p{L}\p{N}_\s]/gu` so umlauts/ß/accented letters stay inside their word instead of fragmenting mid-word — issue #55). ### Entry point

[extension.ts](src/extension.ts) — `activate()` instantiates the `EnablementManager` first (so the decorator + CodeLens provider can be handed its `isEnabled` predicate), then wires the edit listeners + commands, registers the CodeLens provider (`languages.registerCodeLensProvider` for `file`/`untitled` schemes) and the `CriticHoverProvider` (`languages.registerHoverProvider`, same schemes) — which one actually shows actions is decided at request time by `kaicrit.edit.changeActions` (default `hover`), creates the changes Tree View (`window.createTreeView('kaicrit.changes', …)`), instantiates the `TrackChangesManager` (its `handleChange` runs first in the shared `onDidChangeTextDocument` listener, before `dm.scheduleUpdate`), registers the compare commands and the double-pane commands (`registerDoublePaneCommands`), and **returns** `{ extendMarkdownIt }` (passing the current `kaicrit.edit.commentMetadata` value into `criticMarkupPlugin`) so the built-in preview picks up the plugin. That `commentMetadata` value is read once and frozen into the cached markdown-it instance — VS Code offers no stable API to rebuild it on a config change, so a `kaicrit.edit.commentMetadata` toggle only reaches the preview after a window reload (documented known limitation in `docs/preview.md`); the editor parser re-reads the value per parse, so decorations need no reload. The `EnablementManager.onDidChange` handler re-runs `dm.update` for every visible editor (and refreshes the status bar + `kaicrit.hasChanges` context) so a per-file toggle or an `enabledLanguages` edit takes effect immediately; `onDidChangeActiveTextEditor`/`onDidCloseTextDocument` also call `em.syncStatusBar`/`em.forget`.

## CriticMarkup Types

The edit parser uses the single `RE_ALL` regex from [core/markers.ts](src/core/markers.ts) — via the `findMarkers(text)` helper (`text.matchAll`, no shared `lastIndex`) — with six capture groups (one per type, two for substitution). `document.positionAt()` converts string offsets to VSCode positions — no manual line/column arithmetic. The preview ([preview/markdownIt.ts](src/preview/markdownIt.ts)) keeps a separate inline tokenizer because it integrates with markdown-it's own parser, but shares the delimiter literals via `MARKERS`.

Each `CriticChange` carries:
- `fullRange` — the entire `{--...--}` span, used for all edits
- `contentRange` — content inside markers (non-substitution types)
- `oldRange` / `newRange` — substitution sub-ranges for separate decoration
- `author?` / `date?` — comment metadata, set only on comments with the optional `@author YYYY-MM-DD:` prefix (and only when `kaicrit.edit.commentMetadata` is enabled). The convention lives in [core/comment.ts](src/core/comment.ts); a comment without the prefix carries neither field and behaves exactly as a plain comment.
- `raw?` — the verbatim marker text (`match[0]`) captured at parse time. `commands.ts`'s `applyAt`/`applyAll` confirm `document.getText(fullRange) === raw` before an accept/reject (after flushing any pending debounced parse via `dm.hasPending`/`dm.update`) so a stale cache whose offsets predate an in-flight edit can't resolve against the wrong span and corrupt the document (issue #52).

Marker characters (`{--`, `--}`, etc.) get a separate dimming decoration (opacity 0.4) via `markerType` in `DecoratorManager`. The five content decorations additionally set `overviewRulerColor`/`overviewRulerLane` (Right) so changes show on the scrollbar; the dimmed `markerType` deliberately omits ruler markers.

Multi-line comments (`{>>line1\nline2<<}`) render fully in the preview: the inline tokenizer matches across newlines via `indexOf`, and `.critic-comment` in [media/critic.css](media/critic.css) sets `white-space: pre-wrap` so the line breaks stay visible.

## Accept/Reject Semantics

Every resolution replaces `fullRange` with a computed string via one `WorkspaceEdit`:

| Type | Accept | Reject |
|---|---|---|
| Deletion `{--T--}` | `""` | `T` |
| Addition `{++T++}` | `T` | `""` |
| Substitution `{~~O~>N~~}` | `N` | `O` |
| Highlight `{==T==}` | `T` | `T` |
| Comment `{>>T<<}` | `""` | `""` |

`acceptAll` / `rejectAll` collect all replacements into a single `WorkspaceEdit` and apply atomically — no offset-drift issues.

## Adding a New Markup Type

1. Add to `ChangeType` in [core/types.ts](src/core/types.ts)
2. Add the delimiters to `MARKERS` and a capture group to `RE_ALL` in [core/markers.ts](src/core/markers.ts), plus a branch in the parse loop in [edit/parser.ts](src/edit/parser.ts)
3. Add a `TextEditorDecorationType` and handle it in `applyDecorations()` in [edit/decorator.ts](src/edit/decorator.ts)
4. Extend the shared `ORDER`/`LABELS`/`SYMBOLS` tables in [edit/statusBar.ts](src/edit/statusBar.ts) so both the status bar and the sidebar group nodes (which reuse `SYMBOLS`/`LABELS` in [edit/changesView.ts](src/edit/changesView.ts)) pick up the new type
5. Add insert command and accept/reject case in [edit/commands.ts](src/edit/commands.ts)
6. Render the new type in [preview/markdownIt.ts](src/preview/markdownIt.ts) (+ a `.critic-*` class in [media/critic.css](media/critic.css)) if it should appear in the preview
7. Add command entry to `contributes.commands` in [package.json](package.json)
8. Add one or more `contributes.colors` entries in [package.json](package.json) for the new decoration color(s)
9. Update README.md, docs/markup.md, and CLAUDE.md
